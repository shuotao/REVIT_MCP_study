核心觀念:雙軸盤點

族群清整最容易出錯的地方,是「只看到一半」。這個 domain 強制你同時看兩個軸:

軸看的是什麼工具
載入軸Project Browser 裡看得到的所有類型（不管有沒有被放進模型）逐族群跑 list_family_symbols
放置軸模型裡真的有實例的類型query_elements

關鍵等式

載入 − 放置 = 未使用類型（purge 對象）

  • 只看放置 → 會漏掉「載入了但沒人用」的未使用類型,永遠 purge 不掉。
  • 只看載入 → 不知道哪些類型真的有人在用,不敢動。
  • 兩軸相減 → 才能安全鎖定 purge 目標。

各階段用哪些工具

階段工具目的
Phase 0get_active_viewre-anchor 當前視圖（活躍狀態重新錨定）
Phase 1query_elements_with_filter + list_family_symbols雙軸盤點:放置 vs 載入
Phase 2get_element_info讀「簽章」（實際參數值）
執行change_element_type / delete_element / modify_element_parametermerge / purge / rename

為什麼要逐族群

list_family_symbols 一次有 100 筆截斷上限。直接全抓會漏。盤點時用逐族群過濾,才不會被截斷吃掉尾巴。

真實使用流程(5 步)

  1. 先讓 Revit 閒置:按 Esc 清掉進行中的命令、關掉對話框。否則 MCP 指令會排隊等 Revit 閒置,30 秒沒輪到就回報「Command timed out」——這不是斷線、也不是埠問題,別去重點 ribbon,把 Revit 放空再重試即可。（原因:命令經 ExternalEvent 在 UI 執行緒執行,Revit 非閒置時不處理。）
  2. 叫用 /family-inventory-cleanup,指定要整理的 category（門 Doors / 窗 Windows / 柱 Columns…）。
  3. 讓它跑 Phase 0 → 1 → 2 → 提案,然後停在提案等你選。
  4. 你勾選要做的（purge / merge / rename）後才執行;每個破壞性動作前會過「強制前置檢查」;破壞性動作逐一執行、不並行;做錯可在 Revit 按 Ctrl+Z 復原。
  5. 執行後重查驗證,輸出 SELF-AUDIT。

逾時不是斷線

看到「Command timed out」是最常見的誤判點。它代表 Revit 當下不在閒置狀態（有對話框、有未結束的命令、滑鼠正在拖曳)。不要重啟服務、不要去戳 ribbon——按 Esc、關對話框,把畫面放空後再下一次指令就會通。

強制前置檢查(7 條硬規則)

每個破壞性動作前都要過這份清單。它們不是建議,是擋掉「靜默誤刪」的護欄。

  1. 盤點用逐族群過濾,避開 list_family_symbols 的 100 筆截斷。
  2. 刪型前驗 0 實例——確認真的沒人在用才刪。
  3. delete_element 會連帶刪實例且不回報,動作前自己先列出影響範圍。
  4. 判重複 / 名實一律用 get_element_info 讀簽章,禁用名稱捷徑。
  5. 逐一執行,不並行——破壞性動作彼此可能互相影響基準。
  6. 告知 Ctrl+Z 可復原——所有動作在 Transaction 內,使用者可在 Revit 撤銷。
  7. 每批前重抓基準——前一批的執行會改變現況,下一批要重新盤點。

陷阱在哪

主力型名為 30"D x 30"W(93 根),但 get_element_info 讀出的實際 Depth/Width = 24" × 24"——名字完全反指。純名稱比對會把這 93 根判成 30×30。

完整修正步驟 + 先導試水

簽章揭穿真重複

30"D x 30"W(實 24×24)與另一個 24"D x 24"W 2(實 24×24)幾何簽章完全相同 = 真重複,其中一個(主力那個)名字是錯的。

修正(柱 Columns)

  1. 把 24"D x 24"W 2 的 1 根用 change_element_type 併入。
  2. 刪掉空出的重複型。
  3. 用 modify_element_parameter(Name 特例)把主力型改名為正確的 24"D x 24"W。
  4. 再 purge 5 個未使用型:10"D x 12"W、10"D x 16"W、15"D x 25"W、17.5"D x 25"W、Round Column 的 12" Diameter。

結果:Rectangular 13 → 8 型,全部名實相符;總柱數 121 → 121,零實例誤刪。

先導試水(門 Door-Opening)

在動柱之前先拿門練手:10 型 → 3 型——刪一個疑誤建的 50" x 60"(連帶 1 實例)、purge 5 個未使用、merge 一個 36" x 84" 2 重複。

這次測試補進 domain 的 5 條硬規則

  1. 未知 category 要先做「族群發現」再盤點。
  2. 簽章強制,禁名稱 / get_category_fields 捷徑——後者在異質類別會抽錯樣本(柱實測抽到 Corinthian,根本沒回 Depth/Width)。
  3. 結尾的「 N」尾碼只是線索,要分辨真重複 / 孤兒 / 名字錯三種情況。
  4. Mark 群聚法只在有 Mark 的類別有效(柱無 Mark)。
  5. filter 字串打錯會回「假 0」,0 實例驗證優先用完整列舉,別只信 filter 回傳的 0。

殘留風險(再次提醒)

這次依「24×24 為真、30×30 是名字打錯」執行。但簽章只能揭露「名≠參數」,不能決定設計本意。若本意其實是 30×30,那這 93 根是斷面不足的結構錯誤,改名只是把問題藏起來——需結構技師確認,超出 MCP 能判範圍。
