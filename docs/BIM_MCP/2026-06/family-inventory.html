DUAL-AXIS

核心觀念：雙軸盤點

族群清整最容易出錯的地方，是「只看到一半」。同時看兩個軸才看得到 purge 對象。

載入軸

Project Browser 看得到的所有類型（不管有沒有被放進模型）。逐族群跑 list_family_symbols（一次 100 筆截斷上限，要逐族群過濾才不漏）。

放置軸

模型裡真的有實例的類型。query_elements。只看放置會漏未使用；只看載入不知道誰在用。

載入 − 放置 = 未使用類型（purge 對象）

7 MANDATORY RULES

每個破壞性動作前的「強制前置檢查」

不是建議，是擋掉「靜默誤刪」的護欄。

1盤點用逐族群過濾，避開 list_family_symbols 的 100 筆截斷
2刪型前驗 0 實例——確認真的沒人在用才刪
3delete_element 會連帶刪實例且不回報，動作前自己先列影響範圍
4判重複／名實一律用 get_element_info 讀簽章，禁名稱捷徑
5破壞性動作逐一執行、不並行（彼此會改變基準）
6告知 Ctrl+Z 可復原（所有動作在 Transaction 內）
7每批前重抓基準（前一批執行已改變現況）
REAL CASE · COLUMN

名字寫 30、其實是 24 的主力柱

全專案 121 根柱、4 族群。主力型名為 30"D x 30"W（93 根），但 get_element_info 讀出 Depth/Width 實際是 24" × 24"——名字完全反指。純名稱比對會把這 93 根誤判成 30×30，這就是「名稱會騙人」的鐵證。

簽章揭穿真重複 → 修正

30"D x 30"W（實 24×24）與 24"D x 24"W 2（實 24×24）幾何簽章相同 = 真重複。把後者 1 根用 change_element_type 併入 → 刪空出的重複型 → 用 modify_element_parameter（Name 特例）把主力型改名為正確的 24"D x 24"W。

結果

再 purge 5 個未使用型。Rectangular 13 → 8 型，全部名實相符；總柱數 121 → 121，零實例誤刪。動柱前先拿門練手：Door-Opening 10 型 → 3 型。

RISK · BOUNDARY

殘留風險與能力邊界

改名不等於修對

本次依「24×24 為真、30×30 是名字打錯」執行。但簽章只能揭露「名 ≠ 參數」，不能決定設計本意。若本意其實是 30×30，那這 93 根是斷面不足的結構錯誤，改名只是把問題藏起來——需結構技師確認，超出 MCP 能判範圍。

能力邊界

做得到：盤點 · 簽章比對 · 改參數值 · 換類型 · 單型改名（Name 特例，可 Ctrl+Z）· 刪除類型。
做不到（需 Family Editor 或別解）：改參數資料格式（Text↔Number）· 改族群內部定義 · 結構柱 OST_StructuralColumns。
