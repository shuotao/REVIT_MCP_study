Skills · 編排層
何時觸發 / 什麼順序

Skills 是編排（orchestration），不是知識本身。一個 Skill 規定「什麼關鍵字觸發 / 呼叫哪些工具 / 套用哪個 Domain SOP」。Skill metadata 永遠常駐於 AI Client context；body 與 Domain 被引用時才載入。本頁列 42 個 Skill 的描述、觸發關鍵字與對應 Domain。

FRAME · 三層架構

Skill / Domain / Tool 三者分工

Skill（編排）

何時觸發 + 呼叫哪些 tool + 什麼順序。42 個。

Domain（知識）

法規條文、SOP、計算公式。61 個。被 Skill 引用。

Tool（執行）

原子操作（query / create / modify）。146 個 MCP tools。

「不要把每個 Domain 都升級成 Skill」——這是本專案的核心架構原則。詳見 22 命題。

完整三層定義見 三層架構 V2。

RATIONALE · 為什麼不每個 Domain 都升級成 Skill

42 Skill vs 61 Domain — 為什麼數量不同？

❌ 反例：把每個 Domain 都升 Skill

40 Skill metadata 全部常駐 → AI context window 被吃掉一大塊 → 真正需要時反而找不到。

且重複邏輯——防火 SOP 同時被消防、走廊、合規三個 Skill 引用，搬到各自 Skill body 內就重複三次。

✓ 本專案做法

Skill = 編排（42 個 metadata 常駐）；Domain = 知識（61 markdown，被引用才載入）。

知識複用，編排薄層。一個防火 SOP 被三個 Skill 共享，原文只有一份。

原則
Skill 加入門檻高：必須有跨多 tool 的編排價值且觸發頻率高到值得常駐 metadata。否則就讓它停在 Domain。
TECHNIQUE
Orchestration vs Knowledge Separation · Metadata Footprint Control · Knowledge Deduplication
CHECKLIST
  • 新 Skill 是否協調 ≥ 2 個 tool？單一 tool 通常不必開 Skill。
  • 觸發關鍵字是否高頻？冷門關鍵字停在 Domain 即可。
  • 是否引用既有 Domain？不寫新知識，寫新編排。
